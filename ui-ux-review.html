01 — App Overview

Purpose & Target User

Understanding the context before judging design choices.

📦 What this app is

MB.OS V-Cycle AI Orchestrator is an internal engineering tool for MBRDI ECU (Electronic Control Unit) software engineers. It automates the software quality lifecycle — Static Code Analysis (Astrée) → Unit Testing (Tessy) → Integration — and uses AI (Gemini) to generate analysis reports, summaries, and recommendations at each gate.

🎯 Assumed Primary User

An embedded software engineer or team lead at MBRDI who manages one or more ECU feature modules. They understand AUTOSAR, C/C++, static analysis, and test coverage — but are not necessarily UX-savvy. They need to quickly see which modules are failing, why, and what to do next.

✅ Design Intent — Well Aligned

The app correctly adopts a pipeline management metaphor (stages, statuses, gate-passing). The card + table layout, status badges, and multi-step modal are all appropriate patterns for this kind of tool. The overall visual language is professional, not consumer, which matches the audience.

Critical

Two-step to start: Modal + separate Play button

After creating a module, the user is dropped back to the dashboard and must find the Play button to actually start the V-Cycle. The modal's final "Add Module" CTA implies completion, but the workflow hasn't started. This creates a gap in user intent — they came to run a module, not just register it.

→ Fix: Add a "Save & Start V-Cycle" primary action on Step 3, with "Save Only" as secondary.
Major

No progress indicator in the multi-step modal

Users on Step 1 have no visual awareness that there are 3 steps total. There's no stepper component, no "Step 1 of 3" label, nothing. This causes disorientation and reduces confidence on first use.

→ Fix: Add a simple 3-dot or numbered stepper below the modal header.
Major

ECU Target field is not wired up

The "ECU Target" dropdown on Step 1 has no value or onChange handler. The selection is silently discarded. This is a logic bug masquerading as a UI issue — it will confuse engineers who set this field expecting it to matter.

→ Fix: Wire up the field to newFeature state and surface the selected target in the Step 3 review panel.

Major

RotateCcw icon is ambiguous — "Retry" isn't obvious

The retry action for failed modules uses a RotateCcw (circular arrow) icon with no label. In different tools this icon means "undo," "refresh," "retry," or "reset." Without a visible label or strong tooltip, users may hesitate or misinterpret the action — especially damaging in a tool where accidental re-runs have cost implications.

→ Fix: Add a visible text label "Retry" next to the icon, or use a labelled contextual dropdown for actions.
Major

No failure summary visible on the dashboard

A red "Static Code Analysis Failed" badge tells you that it failed, but not why — even at a glance (e.g. "12 warnings, 0 errors"). Engineers often need to decide priority from the table view without clicking in. The Checks column shows "4 / 6" but this doesn't communicate failure reason.

→ Fix: Surface the top-level failure metric in the row — e.g. a mini inline stat "12 warnings" on hover or as a sub-label.
Positive

Contextual action system is smart

The renderActions() function shows different CTAs based on module status (Play for ready, Retry for failed, Report for complete). This is exactly the right pattern — it reduces cognitive load by surfacing only the relevant next action.

04 — Screen-by-Screen Review

Component Analysis

Detailed findings for each major view and component.

Global Navigation (Layout.tsx)
⚠ Incomplete
Critical

3 of 4 nav links are dead (#)

"Homepage", "V-Cycle Module", and "Settings" all link to # and are indistinguishable from the live "Dashboard" link. Users clicking these links get no feedback, no navigation, and no indication the feature isn't available. In a demo context, this will erode trust immediately.

→ Fix: Visually disable unimplemented links (muted color, cursor-not-allowed, tooltip "Coming soon"), or remove them from the nav until built.
Major

Bell and Search icons are imported but never rendered

The Bell and Search icons are imported in Layout.tsx but not rendered in the final JSX. If these are planned features (notifications, global search), they should be deferred properly or added as visual placeholders with disabled state. If not planned, remove the imports.

Minor

No breadcrumbs when inside a V-Cycle Workflow

When navigating to a module's V-Cycle view (/workflow/:featureId), the nav shows the same top-level links with no indication of where the user is in the app hierarchy. A back link or breadcrumb ("Dashboard > MonEmmBatTm > Static Analysis") would prevent disorientation.

→ Fix: Add a breadcrumb component beneath the navbar on non-root routes.
Positive

Floating Copilot launcher is well-designed

The fixed-position Copilot button with smooth scale/opacity animation, dark glass-panel header, and toggle-to-close logic is a polished interaction. The full-height right-panel layout for the chat is appropriately sized (400×600px). The Sparkles icon correctly communicates "AI" in modern product language.

Dashboard (Dashboard.tsx)
⚠ Needs Work
Critical

Page content clips on short viewports — no scroll

The outer Layout sets main to overflow-hidden, and the Dashboard inner container has no overflow-y-auto. On a 768px-height screen, the stat cards + tabs + table overflow beyond the viewport with no scrollbar. All content below the fold is unreachable.

→ Fix: Change Dashboard root div to h-full overflow-y-auto and ensure the parent Layout's main is overflow-hidden (already done) or overflow-y-auto.
Critical

Status badge colour semantics: Amber for "Passed" is wrong

STATIC_ANALYSIS_PASSED and UNIT_TESTING_PASSED render amber badges with a Clock icon. Amber universally signals warning/in-progress, and a clock icon implies waiting — not success. An engineer scanning the table may act on these as if something is wrong when it isn't, creating false urgency.

→ Fix: Use blue badges with a ChevronRight or ArrowRight icon for "passed but pipeline continues" states. Reserve amber for truly indeterminate/warning states.
Major

"Validation Tasks" tab shows static count, renders nothing

The tab shows a hardcoded 2 badge and switching to it produces no visual change. A user exploring the interface will click this and see no result — blank or identical — which signals a bug rather than an empty state. This actively damages first impressions.

→ Fix: Implement an empty state ("No validation tasks yet. Tasks are created automatically when a module fails a gate.") or hide the tab until the feature exists.
Major

Coverage column has no context labels

The two badge chips (88% / 92%) have no visible labels. What does each represent? "Actual / Target"? "Branch / Statement"? This is ambiguous for anyone unfamiliar with the data model. The Feature type defines it as just a string — the intent is not documented or surfaced.

→ Fix: Add a column tooltip or sub-label pair: "Actual" and "Target" micro-labels beneath each value. Alternatively, show as a progress bar with target marked.
Minor

Filter / Sort / Date Filter buttons are inert

The three toolbar buttons are visible and interactive-looking but have no event handlers. Clicking them produces no response. This is fine as scaffolding at early exploration, but should be either stubbed (e.g. show a "Coming soon" tooltip on hover) or removed to avoid confusion in reviews.

Minor

Table row hover is too subtle

hover:bg-gray-50/50 produces a barely-perceptible highlight. Combined with the already light table background, the hover state is essentially invisible. Users get no tactile confirmation that rows are interactive/clickable. The group class on rows is unused.

→ Fix: Increase to hover:bg-gray-100 or add a left-border highlight on hover. Make rows clearly clickable (consider routing to the workflow detail on row click, not just the action button).
V-Cycle Workflow (VCycleWorkflow.tsx)
✓ Mostly Good
Positive

3-stage Stepper with scale + shadow on active step is excellent

The stepper correctly uses scale-110 and a stronger shadow for the active step, plus bg-emerald-500 with a checkmark for completed steps. This is a clean, readable pipeline indicator. The connecting line animating from gray to emerald is a polished touch.

Positive

File Viewer (IDE-style) is a strong design choice

Showing the actual source files (acc_logic.c, test files, integration logs) in a VS Code-inspired file explorer + dark code panel gives engineers the "evidence" they need without leaving the tool. The file type icons (Code2, FileText, FileJson) with colour coding are well-differentiated.

Major

Analysis auto-starts on mount with no user confirmation

In the useEffect, if the module status is READY_FOR_STATIC_ANALYSIS or STATIC_ANALYSIS_FAILED, startAnalysis() is called immediately. This means navigating to the workflow URL immediately triggers the Gemini API — no "Start Analysis" confirmation step. This is both a UX issue (no user control) and a cost/safety concern (unintended API calls).

→ Fix: Show an "idle" state with a clear "Run Analysis" CTA on first load, rather than auto-triggering on mount.
Minor

Stepper step labels use ALL-CAPS tracking-widest — hard to read at small size

The step label typography (text-[10px] font-bold uppercase tracking-widest) is positioned with absolute -bottom-8 which risks overlapping on narrower screens. "Static Code Report" with wide tracking could collide with "Unit Testing".

→ Fix: Use sentence case or title case labels at 11–12px, and test at 375px viewport width.
Summary Tool (SummaryTool.tsx)
⚠ Needs Work
Major

Tool exists as a standalone route but is unreachable from the main nav

SummaryTool lives at /summary but no navigation link, breadcrumb, or card in the dashboard points there. Same applies to TestCaseTool, TestQualityTool, VisualizationTool, UncoveredBranchesTool, AstreeTool, and HistoryTool. Eight screens exist with no navigational entry point — they're invisible to users unless they know the URL.

→ Fix: Determine whether these tools surface within the VCycle workflow steps (as sub-panels) or as standalone pages navigable from a side nav. Either is valid, but the decision must be made and implemented.
Minor

Traceability warning is visually buried

The mock data includes an important warning: "Potential wrong linking of requirement id to the test case: PV_EMM-87779." For automotive software, traceability errors are critical (ISO 26262 / ASPICE compliance). This warning should be front-and-centre, not a secondary piece of data.

→ Fix: Surface traceability warnings as a prominent alert panel at the top of the summary view, not inline with other data.

06 — Information Architecture

Structure & Navigation

How the app organises content and how users find their way around.

🚨 Major IA Gap — 8 Routes with No Navigation

The app defines 10 routes but the navbar exposes access to 1 (Dashboard). Eight tool components (SummaryTool, TestCaseTool, TestQualityTool, VisualizationTool, UncoveredBranchesTool, AstreeTool, HistoryTool, and the VCycleWorkflow) are completely invisible from the navigation layer. This suggests the intended IA hasn't been built yet — the routing exists but the navigation shell doesn't match it.

⚠ Decision Needed: Flat nav or nested within workflow?

There are two valid IA models for this tool. Option A (Flat): Add a left sidebar with all tools listed, accessible independently from any point. Option B (Nested): Tools surface contextually within the VCycle workflow steps (e.g. clicking "Unit Testing" step opens the TestCaseTool inline). Option B is more sophisticated and appropriate for a pipeline tool — but it requires the workflow component to orchestrate tool display.

💡 Recommended IA Structure

Given this is an automated pipeline tool, the recommended structure is: Dashboard (module registry + stats) as the home, with the V-Cycle Workflow as a full-screen detail view per module. Within each workflow step, the relevant tools (Astrée, TestCase, Summary etc.) should surface as sub-panels or expandable sections — not separate top-level routes. The dedicated tool routes could remain for standalone/deep-link access but shouldn't be primary navigation.

08 — Accessibility Audit

WCAG 2.1 AA Review

Accessibility findings against the four WCAG principles: Perceivable, Operable, Understandable, Robust.

❌ Perceivable

  • Icon-only action buttons lack visible text; title attribute is not a reliable accessible name
  • Coverage column provides no textual context for screen readers ("88%" is meaningless without "actual coverage")
  • Status badge icons (AlertCircle, CheckCircle, Clock) are purely decorative SVGs with no aria-label
  • focus:ring-indigo-500/20 — 20% opacity focus ring fails contrast in all backgrounds

⚠ Operable

  • Table rows are not keyboard-navigable (no tabIndex, no row click handler)
  • Modal cannot be dismissed with Escape key (no onKeyDown handler visible)
  • Copilot floating panel: focus not trapped when open, focus does not return to launcher on close
  • No skip-to-main-content link

⚠ Understandable

  • Amber "Passed" badges will confuse both sighted users and AT users — inconsistent state semantics
  • No lang attribute on <html> in index.html (screen reader language determination fails)
  • Inline text-[10px] label text may be below minimum readable size for low-vision users

✓ Robust

  • Semantic HTML used correctly — <table> with <thead>, <th>, <tbody>
  • React with TypeScript ensures type safety in rendered props
  • Inter + JetBrains Mono are well-supported, readable fonts
  • Tailwind produces valid, predictable CSS
🔴 Highest Priority A11y Fix

Add aria-label attributes to all icon-only buttons. This is a one-line fix per button and is the single highest-impact accessibility improvement. Example:

// Before
<button className="..." title="Start V-Cycle">
  <Play size={18} />
</button>

// After
<button className="..." aria-label="Start V-Cycle for {feature.name}">
  <Play size={18} aria-hidden="true" />
</button>
09 — Visual Design

Design System Observations

What's working, what needs codifying, and what's inconsistent.

Colour Palette
Partial
Positive

Indigo-800 primary is a strong, distinctive choice for engineering tooling

Indigo-800 (#3730a3) sits between consumer-blue and enterprise-navy. It feels technical without being cold. Consistent use across buttons, active states, links, and the navbar logo creates strong visual coherence.

Critical

Semantic colours are not fully consistent

Amber is used for both pipeline-in-progress (intended) and analysis-passed (wrong). Red correctly maps to failure states. Emerald maps to success/complete. Blue is unused as a semantic colour but would be the natural fit for "passed, progressing." Define a formal 5-colour semantic system: Indigo (action/primary), Blue (in-progress/awaiting), Amber (warning/needs attention), Red (error/failed), Emerald (success/complete).

Typography
Good
Positive

Inter + JetBrains Mono is the right pairing

Inter's proportional spacing works well for dense data tables. JetBrains Mono for branch names (feat/battery-monitor) and code views creates an immediately legible context switch. The font-size scale from text-[10px] to text-3xl is broad but each usage feels intentional.

Minor

text-[10px] labels may be too small

Several labels use text-[10px] (modal field labels, table headers). At non-retina screens, this can read as blurry or too small for prolonged use. Minimum recommended body text is 12px; labels at 11px are acceptable. Drop all text-[10px] uses to text-xs (12px).

Spacing & Border Radius
Inconsistent
Major

Border-radius scale is not codified

Current usage: rounded-3xl (stat cards, table container), rounded-2xl (modal, file viewer), rounded-xl (inputs, selects), rounded-lg (buttons, action buttons), rounded-full (badges, copilot button). This 5-level scale is reasonable but should be explicitly defined: Container = 3xl, Panel = 2xl, Input/Card = xl, Button = lg, Pill/Badge = full. Currently it's pattern-matched rather than principled.

✅ Summary Verdict

This is a genuinely strong early exploration. The visual language is coherent, the core metaphor (V-Cycle pipeline) is correctly chosen, and several interaction decisions (contextual actions, file viewer, AI analysis sections) show real product thinking. The gaps are mostly structural completeness (empty states, error handling, navigation coverage) and a few critical semantic mistakes (amber for "passed", auto-start on mount). None of the issues are architectural — they're all fixable within a focused 1–2 week sprint without redesigning anything from scratch.
